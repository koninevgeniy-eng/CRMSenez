# Implementation plan

Дата фиксации: 18.06.2026

План описывает, как переносить согласованный процесс MVP в текущий проект без полной остановки разработки.

## Phase 0. Зафиксировать процесс

Цель: превратить решения рабочей группы в источник правды для разработки.

Изменения:

- добавить `docs/MVP_PROCESS.md`;
- добавить этот план внедрения;
- использовать документы как основу для задач и проверок.

Риск: низкий.

## Phase 1. Модель процесса и статусы

Цель: заменить неявную цепочку статусов на управляемый workflow.

Изменения в коде:

- обновить `EventStatus` в `src/lib/crm-types.ts`;
- обновить `EVENT_STATUSES` в `src/lib/event-policy.ts`;
- обновить подписи и цвета статусов;
- добавить `ownerId` в `Event`;
- добавить `archivedAt` или использовать статус `archived` как финальный бизнес-статус;
- добавить миграцию Prisma.

Новые/обновленные поля:

- `Event.ownerId`;
- `Event.currentVersion`;
- `Event.archivedAt`;
- `Event.cancelRequestedAt`;
- `Event.cancelRequestedBy`;
- `Event.cancelReason`.

Риск: средний.

## Phase 2. Решения согласования

Цель: хранить не только текущий статус, но и каждое решение по этапу.

Новые модели:

```prisma
model EventApproval {
  id         String   @id @default(cuid())
  eventId    String
  version    Int
  stage      String
  decision   String
  comment    String?
  decidedBy  String
  role       String
  department String?
  createdAt  DateTime @default(now())
}
```

Изменения:

- создать API для решений согласования или расширить workflow API;
- сделать комментарий обязательным при возврате на доработку;
- возвращать карточку именно создателю;
- фиксировать каждое решение руководителя методологии, координации и АГД.

Риск: средний.

## Phase 3. Версионность карточки

Цель: согласовывать конкретную версию карточки, а не плавающее состояние.

Новая модель:

```prisma
model EventVersion {
  id           String   @id @default(cuid())
  eventId      String
  version      Int
  snapshotJson String
  createdBy    String
  createdAt    DateTime @default(now())
}
```

Правила первого внедрения:

- новая версия создается при отправке карточки на согласование;
- согласование привязывается к версии;
- критичные изменения после согласования создают новую версию и запускают повторный процесс, кроме изменения дат АГД;
- пользователю показывается текущая версия и последняя согласованная версия.

Риск: высокий.

## Phase 4. Расширенная история

Цель: сделать историю рабочим инструментом контроля процесса.

Изменения:

- расширить текущий `ChangeLog` или добавить `EventChangeLog`;
- фиксировать поле, старое значение, новое значение, пользователя, роль, департамент, этап и версию;
- показывать историю в карточке мероприятия;
- отличать изменение данных от решения согласования.

Риск: средний.

## Phase 5. Бюджет

Цель: привести бюджет к правилам MVP.

Изменения в модели `BudgetItem`:

- `number`;
- `article`;
- `quantity`;
- `unitPrice`;
- `comment`;
- `overrunReason`;

Правила:

- номер, статья, количество, цена, сумма и комментарий обязательны;
- нулевая сумма разрешена;
- отрицательные значения запрещены;
- при факте выше плана показывать предупреждение;
- причина перерасхода обязательна;
- изменение бюджета после согласования отправляет на повторное согласование бюджета.

Риск: средний.

## Phase 6. Календарь и график

Цель: сделать календарь пригодным для работы АГД.

Изменения:

- считать календарными только мероприятия со статусом `calendar_approved` и последующими статусами;
- добавить фильтры: дата, статус, площадка, ответственный;
- ограничить график одним годом по умолчанию;
- предупреждать о конфликте дат;
- после согласования АГД автоматически переводить карточку в календарный статус.

Риск: средний.

## Phase 7. Организация и задачи

Цель: формализовать передачу мероприятия в организацию.

Изменения:

- руководитель организации назначает руководителя мероприятия;
- руководитель организации назначает сотрудников и задачи;
- кнопка "Мероприятие проведено" доступна руководителю мероприятия от организации;
- после проведения карточка возвращается владельцу в методологию.

Риск: средний.

## Phase 8. Документы, отчет и фото

Цель: закрывать мероприятие структурированно.

Изменения:

- добавить документы типа `program`, `report`, `photo_report`;
- хранить фото спикеров отдельно у спикеров;
- закрытие мероприятия доступно только владельцу карточки после согласования фактического бюджета;
- закрытие переводит карточку в `archived`.

Риск: средний.

## Phase 9. Аналитика

Цель: строить аналитику на нормализованных и согласованных данных.

Изменения:

- обновить источники аналитики под новые статусы;
- учитывать длительность прохождения этапов;
- учитывать план/факт бюджета и перерасход;
- учитывать завершенные и архивные мероприятия;
- не строить итоговую аналитику по незакрытым карточкам, если показатель требует финальных данных.

Риск: средний.

## Рекомендуемый порядок внедрения

1. Phase 0: документация процесса.
2. Phase 1: статусы и владелец карточки.
3. Phase 2: решения согласования.
4. Phase 5: бюджетные правила.
5. Phase 6: календарь.
6. Phase 7: организация и задачи.
7. Phase 4: расширенная история.
8. Phase 3: версионность.
9. Phase 8: документы и закрытие.
10. Phase 9: аналитика.

Версионность лучше внедрять после базового workflow и истории, потому что она меняет смысл согласования и требует аккуратной миграции данных.
